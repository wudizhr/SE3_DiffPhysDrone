# 传感器与模型扩展指南

项目后续会测试不同传感器输入和不同模型结构。为了避免训练脚本分叉，新增能力时遵循两个工厂：

- `sensors/`：负责 observation 构造。
- `model/factory.py`：负责模型创建。
- `control/`：负责把模型原始动作转换成环境可执行控制量。
- `rollout/PolicyRunner`：负责策略 rollout 中的模型调用和动作转换。

## 新增传感器

1. 在 `sensors/` 下新增文件，例如 `mid360.py`。
2. 实现一个 `ObservationBuilder` 子类，声明：
   - `name`
   - `dim_state`
   - `depth_pool_kernel`
   - `model_input_keys`
3. 在 `sensors/factory.py` 的 `OBSERVATION_BUILDERS` 注册。
4. 在 YAML 的 `sensor.name` 中选择它。

MID360 当前的状态：

- CUDA 几何后端已导出 `quadsim_cuda.render_mid360`，可从环境几何渲染世界坐标点云。
- 默认仿真参数采用常用 MID360 组合：水平 360 度、垂直 -7 到 52 度、64 条垂直采样线、0.1 m 盲区、70 m 最大距离。
- 建议先按 `N=20000` 点/帧使用，近似 10 Hz 下约 200k 点/秒的常见 MID360 配置。
- 输出约定为 `points: [B, N, 3]` 和 `ranges: [B, N]`；未命中射线写入 `range=0` 和零坐标。
- `sensors/mid360.py` 已实现点云到伪图像的预处理方法：按方位角 `theta` 和相对 Z 轴极角 `phi` 划分角度 bin，每个 bin 取最近距离，空 bin 填 `r_max`。
- `model/mid360_cnn_model.py` 已提供初始策略模型：`mid360_pseudo_image -> CNN encoder`，`state/odom -> MLP encoder`，特征融合后进入 `GRUCell`，再由 `action_fc` 输出动作。当前 `configs/mid360_cnn.yaml` 对应的伪图像输入为 `[B, 1, 12, 60]`。
- MID360 策略的 state 使用完整姿态矩阵：`use_odom=true` 时为 `[local_v(3), local_target_v(3), R.flatten(9), margin(1)]`，共 16 维；`use_odom=false` 时为 13 维。
- `configs/mid360_cnn.yaml` 是 MID360 训练配置模板；默认 `train.num_envs=16`，更适合作为 16GB 显存上的起步值。`checkpoints/mid360_smoke` 仅用于验证推理链路，不是训练好的策略。

建议后续接入训练 observation 时继续明确：

- 点云坐标系和机体系/世界系变换。
- 点云裁剪范围。
- 是否继续使用当前 pseudo-image，或切换 voxel/grid/range image 表达方式。
- 是否与 depth/odom 融合。
- 输出给模型的张量形状。

## 新增模型

1. 在 `model/` 下新增模型文件。
2. 在 `model/factory.py` 的 `MODEL_REGISTRY` 注册。
3. 在 YAML 的 `model.name` 中选择它。
4. 在 YAML 的 `model.action_mode` 中选择动作语义。
5. 确保 checkpoint 的 `config.yaml` 会保存模型名称、输入维度和动作语义。

## 新增动作语义

模型只负责输出原始 action，动作含义由 `control/` 下的 action adapter 解释。当前已有：

- `accel_velocity`：现有 point-mass 策略动作。模型输出 6 维，按局部坐标拆成 `a_pred` 和 `v_pred`，旋转到世界系后转换成 `Env.run()` 使用的加速度控制。该 adapter 保持原训练和可视化行为不变。

后续接入 CTBR 时，建议新增：

1. `control/ctbr_adapter.py` 或扩展 `control/action_adapter.py`。
2. 在 `create_action_adapter()` 注册 `ctbr`。
3. 新模型输出 4 维 `[collective_thrust, body_rate_x, body_rate_y, body_rate_z]` 的归一化动作。
4. 由 adapter 负责把归一化动作映射到真实推力和机体系角速度范围。
5. 训练和 `PolicyRunner` 不再手写动作公式，只调用 `action_adapter.to_control(...)`。

`se3_model` 当前暂不接入训练和 rollout，因为 CTBR 控制策略尚未完成。等 CTBR action adapter 完成后，再打开工厂注册。

## 不建议的做法

- 不要为每个传感器复制一个训练脚本。
- 不要在 `train/main_cuda.py` 中堆叠大量 `if sensor == ...`。
- 不要把传感器预处理写进模型类。
- 不要在训练脚本或可视化脚本中直接手写 `raw_action -> control` 转换。
- 不要让可视化使用一套 observation 逻辑、训练使用另一套 observation 逻辑。
